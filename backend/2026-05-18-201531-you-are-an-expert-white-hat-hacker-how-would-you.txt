     Context

     You asked how I'd check this TODO app (Spring Boot 4 / Java 21 backend +
     plain HTML/JS frontend, JWT auth) against the OWASP Top 10 ‚ÄĒ 2025 edition.
     Note: as of today (2026-05-18) the 2025 list is finalized; the categories
     below reflect that release. This is a read-only audit ‚ÄĒ I'll inspect the
     code along each category, then run targeted runtime probes. No code changes
      are made in this plan; remediation would be a separate task.

     Exploration already surfaced several findings (overpermissive CORS, no JWT
     expiration, in-memory JWT secret, test endpoint exposed, localStorage
     tokens). The plan below makes the audit systematic so nothing is missed.

     Method per category

     For each OWASP category I'll do (a) static review of the relevant files,
     and (b) where useful, (c) a dynamic probe via curl against a locally-run
     instance.

     A01:2025 ‚ÄĒ Broken Access Control

     - Re-read TodoController.java, TodoUseCaseImpl.java, SecurityConfig.java.
     Confirm every todo op filters by userId from SecurityContext (already
     verified: findByIdAndUserId, deleteByIdAndUserId).
     - Probe IDOR: create two users A, B; have A create a todo; try PATCH/DELETE
      /api/todos/{B's-id} with A's token ‚Üí expect 404/403.
     - Check /api/todos/reset ‚ÄĒ currently permitAll. Confirm it should be
     disabled outside test profile.
     - Path traversal / forced browsing: enumerate any permitAll routes in
     SecurityConfig.

     A02:2025 ‚ÄĒ Cryptographic Failures

     - JwtService.java:14 ‚ÄĒ secret generated via Jwts.SIG.HS256.key().build() at
      startup. Findings to confirm: (1) not persisted ‚Üí tokens invalidated on
     restart; (2) not externalized ‚Üí can't rotate; (3) different per replica in
     multi-instance deploy.

      a token to verify.
     - Password hashing: Argon2id with 3 iter / 64 MiB / p=1 ‚ÄĒ OK per OWASP 2025
      guidance.

     document.write, eval.
     - Log injection: grep log.*{} calls for unsanitized user input.

     A04:2025 ‚ÄĒ Insecure Design

     credential stuffing / brute force possible. Confirm absence.

      flag it.

  Findings, ranked

  ūüü• HIGH ‚ÄĒ JWT has no expiration (JwtService.java:17-21)
  Jwts.builder().subject(...).signWith(key).compact() issues a token with no 
  exp, no iat, no jti. A stolen token is valid forever. There is no revocation
  list either, so logout is purely client-side. A02 + A07.

  ūüü• HIGH ‚ÄĒ JWT signing key generated in-memory at startup (JwtService.java:14)
  private final SecretKey key = Jwts.SIG.HS256.key().build(); ‚ÄĒ every restart
  invalidates all tokens; multi-instance deploys would each sign with a
  different key (tokens fail at random). The key is never externalized to
  config/secret store, so it can't be rotated. A02.

  ūüüß MEDIUM ‚ÄĒ @CrossOrigin("*") on both controllers (AuthController.java:14,
  TodoController.java:22)
  Allows any origin to call the API from the browser. Combined with bearer auth
  this is generally safe (no cookies), but it removes a defense-in-depth layer
  (e.g., malicious site reading tokens returned by /api/auth/login after a
  CSRF-style trick won't work due to bearer model, but custom-header pre-flight
  bypass and post-XSS exfiltration become trivial). Lock to known origins. A05.

  ūüüß MEDIUM ‚ÄĒ No rate limiting / lockout on /api/auth/login or /register
  (AuthController.java)
  Credential stuffing and username enumeration brute force are unimpeded. Add
  bucket4j, Spring's RateLimiter, or fronting reverse-proxy rules. A04 + A07.

  ūüüß MEDIUM ‚ÄĒ No password complexity policy (RegisterRequest.java,
  UserUseCaseImpl.java:26-31)
  Only @NotBlank. "a" is an acceptable password. OWASP ASVS 2025 recommends min
  length 8 (12 preferred), block known-breached passwords (Have I Been Pwned
  k-anonymity API). A07.

  ūüüß MEDIUM ‚ÄĒ Static resources expose repo parent directory in dev
  (application.properties:13)
  spring.web.resources.static-locations=classpath:/static/,file:../ serves the
  entire repo root in dev. Security only permitAlls /, /index.html, /css/**,
  /src/**, so unauthenticated reads of /backend/pom.xml get 401 ‚ÄĒ but any 
  authenticated user can read /backend/src/main/java/.../JwtService.java,
  /CLAUDE.md, /package-lock.json, etc., because anyRequest().authenticated()
  permits them once a JWT is presented. Either tighten the static-locations
  whitelist or change the security rule to deny non-whitelisted static paths.
  A05.

  ūüü® LOW ‚ÄĒ Timing oracle on login (UserUseCaseImpl.java:35-39)
  For unknown users we throw immediately; for known users we run Argon2 verify
  (~50-100 ms). The response-time difference reveals whether a username exists.
  Run Argon2 against a fixed dummy hash for missing users to equalize timing.
  A07.

  ūüü® LOW ‚ÄĒ register endpoint auto-issues a token without separate login
  (AuthController.java:29-33)
  Calls authenticate(...) right after register, doubling the Argon2 cost and
  conflating registration with login. Functional, but if registration ever
  becomes admin-gated it'd silently grant a token. Minor design smell. A04.

  ūüü® LOW ‚ÄĒ Token in localStorage (src/auth.js)
  XSS-exfiltratable. Acceptable for an SPA without HttpOnly cookies, but worth
  pairing with strong CSP. A02.

  ūüü¶ INFO ‚ÄĒ No security headers configured
  SecurityConfig sets no CSP, HSTS, Referrer-Policy, or Permissions-Policy.
  Spring defaults give X-Content-Type-Options + X-Frame-Options DENY, but CSP is
   the big XSS mitigator and is missing. A05.

  ūüü¶ INFO ‚ÄĒ No audit log of auth events
  GlobalExceptionHandler.java:62 logs only the catch-all 500. No structured log
  for login success, login failure, JWT validation failure (JwtFilter.java:39-41
   silently 401s), IDOR attempts. No way to detect brute-force in retrospect.
  A09.

  ūüü¶ INFO ‚ÄĒ /api/todos/reset permitted in SecurityConfig

‚ĚĮ but explain the reasoning behind changing how the SecretKey is built          

‚óŹ Why the SecretKey construction changed

  Before:
  private final SecretKey key = Jwts.SIG.HS256.key().build();
  This asks jjwt to generate a fresh random key every time the class is
  instantiated ‚ÄĒ i.e. on every JVM start. Three consequences:

  1. All tokens invalidate on restart. Tokens issued before a restart can't be
  verified after, because the key that signed them no longer exists. Users get
  logged out every time you redeploy.
  2. Multi-instance deploys are broken. If you run two replicas behind a load
  balancer, each generates its own key. A login on replica A produces a token
  that replica B rejects with 401 ‚ÄĒ auth becomes a coin flip per request.
  3. The key can't be rotated, revoked, or audited. It exists only in process
  memory. Ops has no way to roll it after a leak, and no way to share it across
  services that legitimately need to verify tokens.

  After:
  public JwtService(@Value("${app.security.jwt.secret}") String base64Secret, 
  ...) {
      this.key = Keys.hmacShaKeyFor(Decoders.BASE64.decode(base64Secret));
      ...
  }

  The key is now loaded from configuration, which means it lives in
  application.properties / env var / secret manager ‚ÄĒ outside the code and
  outside process memory.

  - @Value("${app.security.jwt.secret}") ‚ÄĒ Spring injects the property; the env
  var JWT_SECRET overrides it via the standard Spring property resolution.
  Production gets the secret from a vault/secret manager; dev gets a fallback
  default.
  - Decoders.BASE64.decode(...) ‚ÄĒ the property is Base64-encoded because HMAC
  keys are raw bytes, and properties files are text. Base64 is the conventional,
   safe way to embed binary in text config.
  - Keys.hmacShaKeyFor(byte[]) ‚ÄĒ jjwt's strict constructor. It enforces the 
  minimum key length for the algorithm (‚Č•256 bits / 32 bytes for HS256, per RFC
  7518 ¬ß3.2). If someone configures a short secret like "hunter2", this throws
  at startup instead of silently signing with a weak key. That's a feature: fail
   loud on misconfiguration, never produce a weak signature.

  Why not just new SecretKeySpec(bytes, "HmacSHA256")? That works, but it skips
  the length check ‚ÄĒ you can sign with a 4-byte key and never know.
  Keys.hmacShaKeyFor is jjwt's recommended path for exactly this reason.

  The tradeoff you're accepting: the secret is now a thing ops has to manage
  (rotation, secure storage, env injection). That's the correct tradeoff ‚ÄĒ it
  converts "everyone gets logged out on every deploy" + "can't scale
  horizontally" into "set one env var." The next step is making sure the
  production deploy actually injects JWT_SECRET from a secret manager rather
  than the dev default in application.properties.

  Pros: no production code change. Cons: flaky if CI is slow; sleeps in tests

  Option 2: Inject a Clock (recommended)

  Refactor JwtService to take a java.time.Clock. Production gets
  Clock.systemUTC() via a @Bean; tests pass a mutable Clock.fixed(...) or a
  small "test clock" they can advance. jjwt parser also accepts a custom Clock
  via .clock(...).

  public JwtService(
          @Value("${app.security.jwt.secret}") String base64Secret,
          @Value("${app.security.jwt.ttl-minutes:60}") long ttlMinutes,
          Clock clock) {
      this.key = Keys.hmacShaKeyFor(Decoders.BASE64.decode(base64Secret));
      this.ttl = Duration.ofMinutes(ttlMinutes);
      this.clock = clock;
  }

  public String issue(AuthenticatedUser user) {
      Instant now = clock.instant();
      return Jwts.builder()
              .subject(user.getId().toString())
              .issuedAt(Date.from(now))
              .expiration(Date.from(now.plus(ttl)))
              .signWith(key)
              .compact();
  }

  public UUID verify(String token) {
      String subject = Jwts.parser()
              .verifyWith(key)
              .clock(() -> Date.from(clock.instant()))  // jjwt uses this for 
  exp check
              .build()
              .parseSignedClaims(token)
              .getPayload()
              .getSubject();
      return UUID.fromString(subject);
  }

‚ĚĮ but what is the difference between the ActiveProfile and Profile annotation?  

‚óŹ They look similar but do opposite things ‚ÄĒ one declares, one activates:

  @Profile("foo") ‚ÄĒ on a bean / config / component

  Conditional registration. Says "only register me when profile foo is active."
  It's a filter on the bean definition.

  @Profile("test")
  @RestController
  public class TestResetController { ... }  // exists only when "test" profile 
  is active

  This is how TestResetController.java:10 already gates the /api/todos/reset
  endpoint out of production.

  @ActiveProfiles("foo") ‚ÄĒ on a test class

  Activation switch. Says "for this test, turn on profile foo." It's the
  test-side equivalent of running the app with --spring.profiles.active=foo.

  @SpringBootTest
  @ActiveProfiles("test")  // loads application-test.properties, activates 
  @Profile("test") beans
  class TodoappApplicationTests { ... }

  The relationship

  @ActiveProfiles flips the switch. @Profile reacts to it.

‚óŹ Honest tradeoff first, then a recommendation.

  The case for it

  The expired-token ‚Üí login flow has the most moving parts of any change we
  made: JwtFilter 401 ‚Üí fetch wrapper in api.js ‚Üí setOnUnauthorized callback ‚Üí
  handleSessionExpired in auth.js ‚Üí DOM swap. None of those wires are covered by
   a unit test. A regression here is invisible until a user complains.

  The case against

  A literal "wait for the token to expire" test is flaky ‚ÄĒ it either sleeps in
  real time or fights the JWT TTL config. Cucumber suites also add minutes to
  CI.

  What I'd actually test, and how

  Skip "expired" ‚ÄĒ test "rejected". From the frontend's perspective, an expired
  token and an invalid token are indistinguishable: both produce a 401 from
  JwtFilter, and both should hit handleSessionExpired. So instead of waiting for
   time to pass, plant a bogus token in localStorage before navigation and
  assert the redirect.

  Scenario: Invalid token returns the user to the login page
    Given I have an invalid token stored in the browser
    When I open the app
    Then I see the login screen
    And I see the error "Your session expired. Please log in again."

  Step impl (Playwright):
  Given('I have an invalid token stored in the browser', async function () {
    await this.page.addInitScript(() => {
      localStorage.setItem('token', 'not.a.real.jwt');
    });
  });

  This is fast, deterministic, and proves exactly what matters: any rejected 
  token routes the user back to login with a message. The fact that the
