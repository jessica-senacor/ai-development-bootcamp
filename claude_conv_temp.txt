  Boot web app with hexagonal architecture and a use case driven core?

‚óŹ Here's how I'd design it, keeping the hexagonal boundaries clean:

  Core principle: JWT is an infrastructure detail, not domain

  The domain should know about users, credentials, and authentication outcomes ‚ÄĒ
   not tokens, signing keys, or HTTP headers. JWT is one possible adapter for
  transporting an authenticated identity. If you bake JWT into the use case,
  you've leaked infrastructure into the core.

  Layer responsibilities

  Domain (domain/)
  - User ‚ÄĒ id, username, passwordHash (the hash itself is fine; the algorithm is
   a port concern)
  - UserUseCase (inbound port) ‚ÄĒ register(username, password) and
  authenticate(username, password): AuthenticatedUser. Note: authenticate
  returns a domain object, not a token.
  - UserRepository (outbound port) ‚ÄĒ save, findByUsername
  - PasswordHasher (outbound port) ‚ÄĒ hash(raw), matches(raw, hash). Keeps Argon2
   out of the core.

  Application (application/)
  - UserUseCaseImpl ‚ÄĒ validates credentials via PasswordHasher, returns an
  AuthenticatedUser. Has no idea JWT exists.

  Adapters (adapter/)
  - Argon2PasswordHasher (outbound) ‚ÄĒ implements PasswordHasher.
  - AuthController (inbound) ‚ÄĒ calls userUseCase.authenticate(...), then asks a
  TokenIssuer to mint a JWT from the result, returns TokenResponse.
  - TokenIssuer / TokenVerifier ‚ÄĒ these live in the inbound HTTP adapter, not in
   the domain. They're part of how HTTP transports identity.
  - JwtFilter (inbound) ‚ÄĒ verifies the bearer token, resolves the userId, and
  puts it on Spring's SecurityContext. Downstream controllers read it from
  there.

  The key design question: how does TodoUseCase learn who the user is?

  Two viable options ‚ÄĒ pick one and stick with it:

  1. Pass userId explicitly into every use case method. Controller reads it from
   SecurityContext, passes to todoUseCase.getAll(userId). Pure, testable, no
  Spring leakage into the core. Recommended for hexagonal purity.
  2. Inject a CurrentUserProvider port. Use case calls currentUser.id(). Less
  plumbing in controllers, but adds an implicit dependency ‚ÄĒ the use case now
  relies on ambient request state, which makes tests slightly harder.

  Option 1 keeps the use case a pure function of its inputs, which fits your
  current TodoUseCase signatures best.

  Tradeoff to be aware of

  The "right" hexagonal design adds two ports (PasswordHasher, maybe
  CurrentUserProvider) and an AuthenticatedUser domain type just to avoid
  leaking Spring Security and JJWT into the core. For an app this size that's a
  real cost vs. just letting UserUseCaseImpl import a JWT library. The payoff is
   that swapping JWT for sessions, or Argon2 for bcrypt, is a one-adapter change
   ‚ÄĒ and your use case tests don't need any crypto setup.
